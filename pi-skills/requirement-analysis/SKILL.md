# Requirement Analysis

## 适用场景

- 群里出现新需求、问题、风险、争议或模糊目标时。
- 需要先厘清范围、依赖、前提、角色分工，再决定是否正式沉淀时。
- 信息来自零散聊天，尚不足以直接产出正式文档或任务时。

## 何时读取

- 在 runtime turn 的开头，如果你还不能判断“这是要总结、写文档、拆任务，还是先追问/先分析”。
- 当项目资源、群消息、已有结论之间存在冲突或信息缺口时。

## 输出倾向

- 默认先产出 `summary`，用简洁语言说明你的理解、疑点和建议下一步。
- 当分析已经形成稳定结论时，再转向 `document` 或 `task` 输出。

## 常见错误

- 直接把模糊消息翻译成正式任务，跳过澄清和判断。
- 只看当前群消息，不参考项目绑定文档、任务板或近期正式产物。
- 把猜测当事实写进总结。

## 何时不要用

- 需求已经清晰，且明确要求直接写正式文档或拆任务时。
- 只是单纯汇报当前进展，没有新的问题需要分析时。
