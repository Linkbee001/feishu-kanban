# Task Breakdown

## 适用场景

- 需要把需求、方案、目标拆成可执行事项时。
- 群里已经明确了下一阶段要推进什么，但还没有形成清晰任务列表时。
- 需要给出责任人建议、优先级、依赖和下一步动作时。

## 何时读取

- 当你确认这件事应该进入正式跟进，而不是只做聊天总结时。
- 在已有需求/方案文档或稳定共识基础上，准备输出任务板候选项时。

## 输出倾向

- 正式落板时输出 `task`，并显式带上持久化意图和 `bitable` 目标。
- 如果前置信息不足，先输出 `summary` 说明缺口，而不是硬拆伪任务。

## 常见错误

- 产出含糊、不可执行、没有责任边界的任务。
- 把 runtime todo 当成正式项目任务板。
- 未识别依赖、阻塞或待确认项，就直接落板。

## 何时不要用

- 需求本身仍然模糊，需要先分析或先成文档时。
- 用户只是想知道进展或风险，并没有要创建正式事项时。
