# 飞书项目管理助手建设计划 V2

## 1. 产品目标与边界

### 1.1 文档信息

- 文档名称：飞书项目管理助手建设计划 V2
- 文档版本：V2.0
- 文档状态：建设方案
- 编写日期：2026-04-25
- 适用范围：从当前实现形态到后续平台化演进
- 输入文档：`docs/prd_v1.md`、`docs/dev_plan_v1.md`
- 补充方案：`docs/pimono_multi_agent_session_plan.md`

### 1.2 产品定义

飞书项目管理助手是一套以飞书群为项目工作空间、以 pi mono 智能体为协同执行核心、以飞书文档和多维表格为沉淀载体的项目协同系统。

它服务于“一个项目一个群”的研发协作方式，面向项目负责人、产品经理、研发负责人、开发者和测试人员，帮助团队完成：

- 项目初始化与群绑定
- 项目文档沉淀与管理
- 任务规划、任务拆解与进展同步
- 项目进展查询与风险汇总
- 后续基于项目环境的开发任务管理和研发协同

### 1.3 产品目标

本产品的近期与长期目标分为两层：

#### 近期目标

- 用户在新建项目群后，可直接通过群内对话完成项目初始化
- 智能体能稳定识别项目上下文，并把结果沉淀为文档、任务和文件
- 同一个群内的智能体上下文保持一致，避免多人同时消息导致会话混乱
- 不同项目群之间互不影响，可并行处理各自消息

#### 中长期目标

- 智能体从“对话机器人”升级为“项目协同代理”
- 智能体持续参与项目规划、任务推进、风险管理和文档维护
- 智能体在受控前提下扩展到开发任务管理、代码分析与实施建议
- 产品形成可持续扩展的平台型项目协作能力

### 1.4 解决的问题

本产品重点解决以下问题：

- 群聊中的项目上下文无法稳定映射到正式项目对象
- 讨论结果难以自动沉淀为文档和任务
- 多人同时与机器人互动时，回复和上下文容易混乱
- 云端智能体参与研发协作时，缺乏清晰的环境边界和执行模型
- 项目群、项目资源、环境、执行记录之间缺乏统一追溯

### 1.5 当前不纳入默认能力的范围

以下能力不作为当前默认开放能力，只作为后续受控扩展方向：

- 自动代码提交
- 自动创建 PR
- 自动发布上线
- 无确认的批量高风险写入
- 多群共享同一个正式项目上下文
- 多智能体自由竞争同一群上下文

## 2. 核心模型

### 2.1 核心业务模型

本系统采用以下核心模型作为正式基线：

```text
一个项目 = 一个飞书群 = 一个独立智能体会话
```

这意味着：

- 一个正式项目默认绑定一个飞书群
- 一个飞书群默认只绑定一个正式项目
- 一个飞书群对应一个独立的 pi mono 智能体会话
- 不同群拥有不同的智能体会话，彼此并行

### 2.2 会话模型

系统中的智能体会话不是“每条消息一个临时执行”，而是“每个群一个持久逻辑 Agent”。

正式约束如下：

- `sessionKey = feishuChatId`
- 同群消息只能串行进入该群智能体
- 不同群的消息可并行进入各自智能体
- 私聊不复用群会话
- 私聊必须显式选项目，不自动继承最近项目群
- 智能体忙碌时，新消息不进入推理上下文，只返回忙碌提示

### 2.3 角色分工模型

#### 后端负责平台能力

- 飞书事件验签、解密、去重
- 项目、群、环境绑定
- 权限、确认、幂等、审计
- 文档、任务、文件沉淀
- 会话状态记录和执行追溯
- 群级并发闸门

#### pi mono 智能体负责协同能力

- 解析群消息意图
- 判断当前要执行的任务
- 决定调用哪个 skill 或 subagent
- 组织回复内容
- 决定是否输出文档、任务、文件、总结
- 在未初始化群中主动收集初始化信息

### 2.4 项目与环境模型

环境是正式一级对象，不是内部实现细节。

一个项目：

- 至少有一个默认环境
- 可以有多个环境
- 默认环境承接群内大多数正式动作
- 显式环境切换只影响当前群会话

环境至少包含：

- 代码仓地址
- 默认分支
- pi mono 运行标识
- 模型配置
- skill 配置
- 输出目录
- 日志位置

## 3. 总体架构

### 3.1 架构原则

- 飞书是主操作界面，用户尽量不离开飞书完成项目协作
- 后端是平台编排层，而不是主要“思考者”
- pi mono 是群级智能体会话运行底座
- 文档、任务、文件沉淀统一由后端完成
- 所有正式结果都必须可追溯到群、消息、用户、环境和执行

### 3.2 目标态架构

```text
飞书用户 / 项目群
        |
        v
飞书事件回调
        |
        v
NestJS 后端平台层
  |-- Feishu Event Gateway
  |-- Project / Environment Binding
  |-- Group Session Registry
  |-- Confirmation / Permission / Idempotency
  |-- Artifact Sync
  |-- Audit / Trace / Admin
        |
        v
Group Agent Session Adapter
        |
        v
PiMono SDK
  |-- Group Agent
  |-- Skill / Subagent Routing
  |-- Session Memory
  |-- Model Invocation
        |
        v
Structured Outputs
  |-- document
  |-- task
  |-- file
  |-- summary
  |-- log
        |
        v
NestJS Artifact Sync
        |
        v
飞书文档 / 多维表格 / 文件 / 群内消息
```

### 3.3 目标态数据流

#### 场景一：已初始化项目群中的普通消息

1. 飞书事件进入后端
2. 后端验签、解密、去重
3. 后端按 `feishuChatId` 定位项目
4. 后端校验该群是否已有活跃会话锁
5. 若群空闲，则把消息封装为 `SessionMessageEnvelope`
6. 消息进入 `GroupAgentSessionAdapter`
7. pi mono 群级智能体解析任务并执行
8. 智能体输出结构化结果
9. 后端沉淀文档、任务、文件并发送摘要消息

#### 场景二：未初始化项目群中的首次消息

1. 飞书事件进入后端
2. 后端发现该群未绑定项目
3. 后端为该群创建临时初始化会话
4. 群级智能体主动进入初始化信息收集模式
5. 智能体连续追问并汇总项目草稿信息
6. 信息齐备后，后端调用项目初始化流程
7. 自动创建项目、文档目录、任务表、默认环境
8. 群进入正式项目上下文

## 4. pi mono 设计基线

### 4.1 正式运行时基线

正式目标态以 PiMono SDK 为基线，不再以 CLI 文本协议作为长期方案。

采用包：

- `@mariozechner/pi-agent-core`

CLI 仅在迁移阶段保留为 fallback，不写入目标态架构，不作为长期设计前提。

### 4.2 群级智能体适配接口

目标态必须提供如下适配层接口：

```ts
interface GroupAgentSessionAdapter {
  getOrCreateSession(sessionKey: string, context: SessionContext): Promise<GroupAgentSession>;
  submitMessage(sessionKey: string, message: SessionMessageEnvelope): Promise<SessionSubmitResult>;
  isBusy(sessionKey: string): Promise<boolean>;
  getBusyReason(sessionKey: string): Promise<string | null>;
  rehydrateSession(sessionKey: string): Promise<void>;
  closeSession(sessionKey: string): Promise<void>;
}
```

其中：

- `GroupAgentSession` 表示群级智能体运行时对象
- `SessionContext` 至少包含 `projectId`、`environmentId`、`feishuChatId`
- `SessionMessageEnvelope` 至少包含 `messageSourceId`、`senderOpenId`、`rawText`、`sourceType`
- `SessionSubmitResult` 至少区分 `accepted / rejected_busy / failed`

### 4.3 Group Agent 职责

群级 Agent 的职责包括：

- 保持该群的会话记忆和项目上下文
- 对群消息做任务判断
- 在合适时调用文档、任务、总结、分析等 skill
- 在未初始化群中主动切换到“初始化引导模式”
- 输出结构化 `AgentOutput`

### 4.4 SDK 迁移要求

从当前实现迁移到 SDK 时，必须满足：

- 不改变后端的追溯和沉淀接口
- 先保留现有 `PiMonoAdapter` 抽象
- 用 SDK 实现新的 `GroupAgentSessionAdapter`
- 在迁移期保留 CLI fallback 开关
- 完成后逐步去除基于子进程 stdout 的主要依赖

### 4.5 输出契约

不论采用 SDK 还是迁移期 fallback，智能体对后端仍输出统一的结构化结果：

```ts
type AgentOutputType = 'document' | 'task' | 'file' | 'log' | 'summary';
```

后端只接受结构化输出，不以自然语言回复直接写飞书资产。

## 5. 会话与并发控制

### 5.1 正式约束

系统采用“群级会话串行、跨群并行”的并发规则。

#### 同群规则

- 同一 `feishuChatId` 下同一时刻只允许一个消息进入智能体
- 智能体执行期间，新消息不得进入当前群智能体上下文
- 新消息直接返回忙碌提示，不做静默排队

#### 跨群规则

- `chatA` 忙碌不影响 `chatB`
- 每个群有自己的独立智能体会话和入口锁
- 后端与 worker 必须共享同一群级锁视图

### 5.2 群级锁要求

群级锁不能只依赖进程内内存结构，目标态必须使用 Redis 分布式锁或等效分布式闸门。

规则固定为：

- 锁粒度：`feishuChatId`
- 获锁后才能把消息投递给该群智能体
- 未获锁则立即返回忙碌提示
- 不做静默排队
- 锁释放由任务完成、失败、超时、取消统一触发
- API 实例和 worker 实例共享同一把群级锁

### 5.3 会话状态对象

会话状态不只存在于内存，必须持久化为正式对象。

内存中的 live agent 只是运行时实例；真正的会话归属和状态以持久化的群会话对象为准。

### 5.4 忙碌提示规范

同群新消息在智能体忙碌时，系统返回统一语义回复：

- 当前正在处理上一条任务
- 新消息不会打断当前上下文
- 用户可稍后重发

此回复只用于保护上下文，不应被当成失败告警。

## 6. 业务流程

### 6.1 新群首次接入

1. 用户创建飞书群并拉入机器人
2. 用户直接发问题或项目说明
3. 后端发现该群尚未绑定项目
4. 智能体进入初始化引导模式
5. 智能体收集项目名称、简介、仓库、默认分支等信息
6. 后端完成项目、文档目录、任务表、默认环境初始化
7. 群进入正式项目上下文

### 6.2 项目初始化

初始化至少完成以下对象：

- `project`
- 项目文档目录
- 项目多维表格
- 默认环境
- 群级会话注册对象

初始化完成后，群中后续消息默认进入该项目上下文。

### 6.3 普通问答

已初始化群中，用户可直接向机器人发起：

- 需求整理
- 方案生成
- 纪要整理
- 周报总结
- 项目说明问答

若结果适合正式沉淀，智能体输出结构化文档或总结，由后端同步回飞书。

### 6.4 文档沉淀

文档型结果写入项目文档目录，典型类型包括：

- PRD
- 需求分析
- 技术方案
- 会议纪要
- 发布说明
- 周报与复盘

覆盖已有正式文档前必须确认。

### 6.5 任务拆解

任务型结果写入项目任务表，默认流程为：

1. 智能体根据文档或讨论输出任务列表
2. 后端判定为正式写入动作
3. 触发确认卡片
4. 用户确认后写入任务表
5. 每条任务记录来源消息、来源文档、来源环境和来源执行

### 6.6 项目进展查询

用户可直接问：

- 当前项目进展如何
- 现在有哪些阻塞项
- 最近生成了哪些文档
- 本周完成了什么

后端聚合任务表、最近文档、最近执行记录，由智能体负责总结输出。

### 6.7 环境切换

同一项目下可切换环境，例如：

- 默认主环境
- 前端环境
- 后端环境
- 测试环境

环境切换是显式动作，必须确认；切换后只影响该群当前会话。

### 6.8 私聊选项目

私聊不自动绑定群项目。

私聊规则：

- 第一次涉及项目操作时必须选项目
- 私聊上下文只作为临时上下文
- 不允许无确认的正式批量写入
- 私聊不复用任意群级 Agent 会话

## 8. 分阶段建设路线图

### V1：项目协同闭环

目标：完成可用的项目群协同最小闭环。

交付内容：

- 新群接入机器人后，智能体能主动发现“未初始化项目”
- 智能体在群内收集项目名称、简介、仓库、分支等信息
- 自动初始化项目、文档目录、任务表、默认环境
- 群内消息进入当前项目上下文
- 文档型、任务型、总结型输出可沉淀回飞书

### V1.5：群级 Agent 正式化

目标：把当前运行形态从“执行器适配”推进到“群级智能体会话”。

交付内容：

- 将当前 `PiMonoAdapter` 迁到 SDK 基线
- 建立 `group_agent_sessions`
- 引入群级分布式入口锁
- 后端弱化 `IntentMapper`，改为 Agent 主导任务判断
- 未初始化引导、普通问答、文档生成统一走群级 Agent 会话

### V2：项目管理能力增强

目标：从“问答与沉淀”扩展到“持续项目管理”。

交付内容：

- 项目进展查询
- 风险项汇总
- 周报、纪要自动化
- 任务状态机完善
- 环境切换与多环境协作
- 智能体基于项目文档和任务板做持续项目推进

### V3：开发任务管理与研发执行扩展

目标：逐步扩展到受控研发协同。

交付内容：

- 开发任务拆解
- 负责人建议
- 里程碑规划
- 基于项目环境做代码分析、方案生成、实施建议
- 引入 subagent / skill 编排
- 代码修改、PR、交付动作仅作为后续受控能力，不纳入默认开放范围

- 同群忙碌拦截

- 新群首次消息触发初始化引导

- 不同群消息可并行进入各自 Agent
- 文档、任务、文件沉淀成功与失败重试

### 9.3 端到端测试

必须覆盖以下真实场景：

1. 新群拉机器人后直接发需求，机器人先引导初始化，再完成项目绑定

3. 群内拆任务并写入任务表
4. 群内查询当前进展

7. 私聊时必须先选项目，不自动继承最近群会话

### 9.4 UAT 与上线验收

上线前必须在真实飞书群中完成：

- 新项目初始化成功
- 项目文档目录可用
- 任务表可用
- 默认环境可用
- 同群串行、跨群并行行为符合预期
- 忙碌提示不导致上下文污染
- 任一飞书产物都能追溯到来源消息、来源环境和执行记录

## 10. 风险与迁移策略

### 10.1 主要风险

#### pi mono 运行时切换风险

从 CLI 迁移到 SDK 时，可能出现：

- 事件流接口差异
- 输出契约不稳定
- 会话重建复杂度上升

应对策略：

- 保留统一适配层
- 分阶段迁移
- 先让 SDK 承担初始化引导和群级消息会话
- 再完全承接普通执行

#### 会话并发风险

若只依赖进程内锁，会在多实例部署下失效。

应对策略：

- 目标态必须采用 Redis 分布式锁
- 群级锁与会话状态必须统一

#### 输出沉淀风险

智能体自由回复若直接写飞书，容易造成结构不稳定。

应对策略：

- 强制结构化输出
- 所有正式写入统一经后端同步

### 10.2 迁移步骤

建议按以下顺序迁移：

1. 保留现有后端数据模型与沉淀接口
2. 引入 `group_agent_sessions`
3. 把群级锁从进程内结构迁到 Redis
4. 引入 `GroupAgentSessionAdapter`
5. 用 PiMono SDK 实现群级 Agent
6. 保留 CLI fallback 开关
7. 验证同群串行与跨群并行
8. 移除主要 CLI 依赖
